## Decisions
- Archived `src/` and `src-tauri/` under `docs/legacy-tauri/` as migration reference instead of deleting them.
- Used `NSStatusItem` and AppKit overlay windows rather than SwiftUI `MenuBarExtra`, because this app needs click-through desktop windows at screen-saver level.
- Keep the status item accessible to VoiceOver users even though the app has no Dock icon or ordinary main window.
- Pause is treated as runtime-only state; mode, intensity, and night background are persisted.
- The build script uses Xcode for the `.app`; `script/build_and_run.sh --verify` also runs `swift build --disable-sandbox` because this environment can hit SwiftPM sandbox-exec failures inside scripts.
- Release verification uses SwiftPM `--disable-sandbox` for the same restricted-shell reason.
- `script/build_and_run.sh --verify` treats LaunchServices bundle launch as best-effort unless `SAKURA_SKY_REQUIRE_BUNDLE_OPEN=1`, because this automation shell can reject the generated bundle with `kLSNoExecutableErr` even when the executable, Info.plist, Mach-O deployment target, and codesign checks pass.
- `script/build_and_run.sh --telemetry` is intended for a normal macOS user session; this automation shell cannot reach log streaming because LaunchServices rejects bundle open first.
- `scripts/notarize_release_zip.sh` deliberately requires a Developer ID identity and a notarytool keychain profile; do not weaken this into ad-hoc notarization or explicit Apple ID/password environment variables.
- `scripts/notarize_release_zip.sh` should not leave canonical final-only evidence behind after partial notarization failure; inspect `.failed` evidence logs for failed attempts.
- `scripts/check_release_evidence.sh` and `scripts/check_publish_readiness.sh` reject stale `.attempt` / `.failed` notarization logs, so rerun candidate/package cleanup after inspecting failed attempts.
- `scripts/check_release_evidence.sh` reruns `scripts/check_zip_contents.sh` on the current ZIP, so ZIP/app drift is checked live, not only through a previous text report.
- `scripts/notarize_release_zip.sh` and `scripts/check_publish_readiness.sh` require `notarytool-submit.log` to contain a strict `status: Accepted` line; do not treat a non-empty notary log or loose `Accepted` mention as enough.
- `scripts/check_publish_readiness.sh` reruns stapler and Gatekeeper validation against the current `dist/Hazakura Wallpaper.app`; do not rely only on old stapler / spctl log files for the final upload decision.
- `scripts/check_release_evidence.sh` and `scripts/check_publish_readiness.sh` also inspect stored notary / stapler / Gatekeeper evidence content, so empty, stale, or failed logs cannot satisfy final release evidence.
- Use `scripts/prepare_release_candidate.sh` as the default local release-candidate gate before any manual publish or notarization attempt.
- Run `scripts/record_visual_qa_acceptance.sh --accepted --checklist-complete --reviewer "Reviewer Name"` only after completing every applicable item in the human visual pass for the notarized final ZIP.
- Use `scripts/check_publish_readiness.sh` as the final public-upload gate after notarization, normal-session bundle-open evidence with operator provenance, and human visual QA acceptance evidence exist; it also rejects legacy Apple ID/password notary environment variables before evidence checks.
- Treat macOS Reduce Motion as a runtime accessibility constraint; do not overwrite the user's saved intensity just because reduced motion is enabled.
